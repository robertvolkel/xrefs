### Data Sources

The platform will pull from multiple data sources:
- **Digikey** (built) — Primary source for technical parametric data, pricing, availability
- **Parts.io** (built — Decision #77) — Accuris gap-fill enrichment: 17 class param maps across 39 families, fills datasheet-only specs that Digikey lacks (thyristor tq/dv_dt, relay coil/contact specs, LDO dropout/regulation, fuse I²t, etc.)
- **Atlas** (built — products + param dictionaries + gaia mapping + description extraction + admin panel + coverage analytics + Explorer QC tool; planned — company profiles) — Chinese component manufacturer dataset: 1,000+ manufacturers in the `atlas_manufacturers` identity table (several hundred actively shipping products; the early "~115" figure is a stale first-ingest count), products in Supabase `atlas_products` (the subset with enough parametric data is scored), 28 L3 family Chinese translation dictionaries + 14 L2 category dictionaries (Decision #102) + 12 family gaia datasheet-extraction dictionaries via shared JSON (`atlas-gaia-dicts.json`, Decision #100). L2 dictionaries map Chinese+English param names to standard L2 attribute IDs for products outside L3 families. `classifyAtlasCategory()` uses c1 guards to prevent cross-domain misclassification (Decision #104) — e.g., laser diodes ≠ rectifiers, RF amplifiers ≠ op-amps, audio connectors ≠ audio devices. Skip list is dictionary-aware (dictionary entries override skip). Gaia mapping handles structured `gaia-{stem}-{Min|Max|Typ}` params with suffix preference and dedup. **Vendor-name hygiene** (Decision #201): the `gaia-` prefix is a vendor-internal detail and must NEVER reach end users — `parseGaiaParam()` strips it at parse time (DB keys are stems, not prefixed), `gaiaId` removed from the `AtlasManufacturer` API response shape so it doesn't ship in network payloads, and `stripGaiaPrefix(s)` belt-and-suspenders sanitizer in `atlasMapper.ts` guards the two fallback construction sites (`mapAtlasModel` raw-param fallback + `fromParametersJsonb` humanize fallback) against vendor format drift. Sanitizer is no-op on every dictionary-mapped value; admin-only `GlobalUnmappedParamsTable` still shows raw paramNames for triage authoring (acceptable carve-out, same as `ManufacturerDetailPage`). LLM description extraction (Decision #112): Claude Haiku extracts structured attributes from product descriptions with quote grounding anti-hallucination — `descriptionExtractor.ts` builds family-aware schema prompts from logic tables, parser rejects any extraction whose source substring isn't found in the original description. Batch script `scripts/atlas-extract-descriptions.ts` runs post-ingest automatically. ~12,510 eligible products, +8pp estimated coverage improvement. LLM description cleanup (Decision #114): `scripts/atlas-clean-descriptions.ts` rewrites raw descriptions into standardized one-liners (max 200 chars) stored in `clean_description` column — runs post-ingest after extraction. `atlasClient.ts` uses `clean_description` when available. AEC qualifications (`aec_q200`/`aec_q101`/`aec_q100`) extracted from parameters JSONB into `part.qualifications` for badge display. Overview tab source badge uses `dataSource` instead of hardcoded Digikey (previously on the now-removed Risk & Compliance tab — Decision #134). Admin dictionary panel with Supabase-backed override layer (Decision #68) — supports both L3 families and L2 categories. Coverage % column + per-family gap analysis drawer comparing Atlas vs Digikey vs logic table (Decision #69). Explorer QC tool: search by MPN/manufacturer with coverage % column, detail drawer with L3 or L2 schema comparison (Decision #102). Admin panel: all manufacturers expandable (scorable + non-scorable), sortable columns, breakdown shows Scorable column. Manufacturer enable/disable toggle (Decision #107): admin can disable manufacturers with poor attribute coverage — disabled manufacturers excluded from search, attribute lookup, and recommendation candidates. `atlas_manufacturer_settings` Supabase table, 60s cached filter in `atlasClient.ts`, optimistic toggle UI in AtlasPanel.
- **atlas_manufacturers table**: Canonical manufacturer identity — 1,011 records from master list with atlas_id, slug, name_en, name_zh, name_display (join key), aliases, partsio_id/name, and JSONB profile columns. Import via `scripts/atlas-manufacturers-import.mjs`. Profile enrichment via `scripts/atlas-api-sync-profiles.mjs` (Decision #139) — syncs descriptions, logos, HQ, certifications, contacts, core products from Atlas external API (`https://cn-api.datasheet5.com`). 297 manufacturers enriched. Env var `ATLAS_API_TOKEN`. Admin detail pages at `/admin/manufacturers/[slug]`.
- **Atlas product flagging**: `atlas_product_flags` table for data quality issues. Flag button in search results and manufacturer Products tab. Flagged tab on ManufacturersPanel and per-MFR detail page.
- **Parameter-aware family reclassification** (Decision #175): `reclassifyByParameterSignals(initial, parameters)` in `lib/services/atlasMapper.ts` runs after `classifyAtlasCategory()` and re-routes B1 products with telltale `Type` parameter values that contradict the c3 verdict — `Bi`/`Uni`/`Bidirectional`/`Unidirectional` → B4 TVS, `Regulator`/`Voltage Regulator` → B3 Zener. Mirrored verbatim into `scripts/atlas-ingest.mjs` (the script carries its own classifier copy). Conservative by design: only fires from B1, only on those specific values; other Type values (Standard/Fast/Ultrafast/blank) leave classification untouched. B3 + B4 dictionaries gained `'type'`/`'类型'` entries so reclassified products' Type parameter is recognized at read time (B3 → `_type` deprioritized, B4 → `polarity` for matching). Retroactive script `scripts/atlas-reclassify-by-type-param.mjs` available as a safety net (dry-run by default; idempotent) — though in practice the misclassified products typically sit in pending ingest batches and self-correct on apply rather than needing a DB rewrite. When this rule list grows past ~5 entries, lift to a structured rule table.

Hard-delete only when all attrs are atlas-tagged; otherwise soft-delete (`status='discontinued'`). New-MFR auto-registration step parses filenames before report generation. AI dict triage (`GlobalUnmappedParamsTable`) calls Claude Haiku via `/api/admin/atlas/dictionaries/suggest` with canonical-vs-invented indicators; accepted overrides are consulted at both ingest time (`scripts/atlas-ingest.mjs → loadAndApplyDictOverrides()`) and read time (`fromParametersJsonb` accepts overrides; `atlasClient` prefetches via `fetchAllDictOverrides()`). - **Triage Batch Accept — star the no-caveat high-confidence rows, tick, accept in one click** (Decision #267, ✅ merged `7ea3780` / PR #18): builds on #265's durable verdicts. A ⭐ marks each Triage row the AI rated `accept` + **high** confidence **with no caveat anywhere in its prose**; starred rows get a checkbox, and a **Batch Accept (N)** toolbar button writes them all in one request (rows flip in place — no refetch, Decision #182 — and a snackbar offers one-click **Undo** of the whole batch). **The star is a spotlight, not a gate**: the engineer hand-ticks every box and all existing on-row warnings still render beside it, which is why there is deliberately NO computed safe-subset / server safe-count / preview modal. Predicate `isStarrableRow` in [lib/services/triageBatchApprove.ts](lib/services/triageBatchApprove.ts) (pure, client-safe, shared by component + tests): accept verdict, high confidence, `!explanationHasCaveat`, writable attributeId+Name, a scope (`dominantFamily || dominantCategory`, #178), still-open (not auto-flagged/parked/actively-mapped — a **reverted** override stays starrable). **The caveat rule is load-bearing**: the AI's `confidence` only ever described the *mapping*, so a "high-confidence" row could still say *"verify whether that 1700 mΩ outlier is a data-entry error"* — the engineer's rule is *anything worth inspecting ⇒ no star*. Fixed at the source (the `/suggest` prompt now folds data caveats into the definition of "high") with `explanationHasCaveat()` as a keyword backstop for the ~3k verdicts generated under the old definition. **Trap:** the first marker list contained `ambiguous`, which substring-matched **"unambiguously"** (the AI's word for *maximum* certainty) and stripped stars off the cleanest rows — certainty words are now excluded outright and a `NEGATION_BEFORE` guard ignores negated hedges ("no need to verify"). *A substring blocklist over prose must be checked against the positive forms of its own words.* **Write path** `POST /api/admin/atlas/dictionaries/batch` mirrors the single-write route (NFC+lower+trim → deactivate-then-insert) plus four batch guards: dedupe within the request (two cosmetic variants collapsing to one `(family_id, param_name)` key would violate the partial unique index and fail the whole insert); **per-family** `.in('param_name', names)`, never a composite multi-tuple `.or()` (PostgREST can't OR tuples); per-row insert fallback on a unique-violation race; and **exactly ONE** single-flight `invalidateTriageQueueCache()` at the very end (#234 — the recompute is ~20–30s). Undo deactivates by **primary key** `id IN (...)`, never an unindexed `change_reason` scan. No DB migration (star reads the `confidence` already on `row.suggestion.detail`; Undo uses the existing PK). **Sizing reality (measured live July 12):** of 1,114 stored `accept` verdicts, **1,020 were already mapped** — only ~40 remain open, and they're the medium/caveated residue. The star pays off on the ~22.6k *ungenerated* params, not the existing pile: ~40% of a fresh Generate batch returns `accept`, ~85–93% of those star, caveat net pulls ~7%. **"N AI verdicts exist" ≠ "N rows are waiting" — join against what's already mapped before quoting a backlog number.**

- **Triage near-duplicate clustering — two-tier hybrid + cross-scope mode** (Decision #208, May 29 + amended May 31, 2026): The Triage queue's "+N similar" cosmetic-variant fanout (Decision #186 part f) was extended with two new tiers to cut engineer Accept clicks per round. **Tier 1 (deterministic)** lives in [lib/services/paramNameSimilarity.ts](lib/services/paramNameSimilarity.ts) — `normalizeParamKey` (lifted from `GlobalUnmappedParamsTable.tsx`) + `isFuzzyMatch` (ASCII-only Levenshtein-1 at min length 5). Catches single-char typos like `propogation_delay` ≡ `propagation_delay`. Hard-gated against CJK code points (`电压_max` vs `电流_max` is distance 1 but means voltage vs current) and against unit-suffix stripping (`电压(V)` and `电压(mV)` must stay distinct so bulk-apply can't propagate a wrong unit). Component's `normalizedMatchesByRow` useMemo got a second pass that merges singleton groups by `isFuzzyMatch`; the existing "+N similar" chip surfaces the merged cluster size. **Tier 2 (AI, opt-in, cross-scope by default per the May 31 amendment)** is a per-row "Find Similar (AI)" icon button next to Accept — opens `ClusterPreviewModal` which posts focal + ALL open in-queue candidates (minus focal/Tier-1-siblings/actively-mapped/unscoped) to `/api/admin/atlas/dictionaries/cluster-suggest` (Sonnet 4.6, max_tokens 8000). Parent pre-sorts candidates exact-normalized-key-first so high-likelihood matches (e.g. every other `AEC-Q101` row across families) rank above the `MAX_CANDIDATES=50` cap. Modal renders a checkbox grid with per-row **scope chip** ("B5 MOSFETs" / "MLCC Capacitors" / etc) so context is never hidden; only **high** confidence is pre-checked in cross-scope mode (medium + low require explicit tick — wider blast radius = higher safety bar). **Generic-term warning banner** fires when focal normalizes to one of [`frequency`, `voltage`, `current`, `capacitance`, `inductance`, `resistance`, `power`, `tolerance`, `type`, `style`, `size`, `level`, `time`, `rate`] (helper `isGenericTerm()`); unit-suffixed forms like `Voltage (V)` explicitly don't trigger it because the unit pins the concept. Sonnet prompt picks up extra hardening when `crossScope=true` AND focal is generic — explicit "default isMatch=false unless evidence is strong" with three worked examples (Frequency, Capacitance, Power). Each Accept fires through the existing `acceptMatchWithPrimaryOverride()` so audit trail + `invalidateTriageQueueCacheAndAwaitFresh()` invalidation behave identically; cross-scope writes go to each match's OWN scope (not the focal's). Cache prefix `atlas-cluster-suggest-v2:` (bumped from v1 when cross-scope shipped) keyed `(scopeKind::scopeKey, focalParamName)`, localStorage 7d TTL. **Tier 3 (eager batch-level pre-cluster)** deferred to BACKLOG until usage tells us whether per-row opt-in is the throughput bottleneck.
- **FAMILY_PARAM_SIGNATURES cooccurrence layer + regex `\b` bug fix** (Decision #207, May 27, 2026): The original signatures used JS regex `\b` for trailing boundaries; `\b` doesn't match between letters and `_` (underscore is `\w`), so `/^hfe\b/i` silently failed on real-world Atlas keys like `hfe_min`/`vceo_v`/`bvceo_v`. Fixed by replacing 9 patterns with `(?![A-Za-z0-9])` negative lookahead. New `requiresAlsoMatching?: RegExp[]` field on `ParamSignature` guards 5 shared-across-families signatures: B6 Ic (cooccur with hfe/vcbo/vceo/vebo — IGBTs spec Ic too), B6 fT (same — RF MOSFETs spec fT), B5 Vgs(th) (cooccur with rds_on — IGBTs have gates too), B5 Qg/Qgs/Qgd (cooccur with rds_on — IGBTs spec gate charge), B7 Vce(sat) (cooccur with vces/eon/eoff/ets — BJTs spec switching saturation). Rds(on) is the only truly MOSFET-unique signal (IGBTs are bipolar conduction). `reclassifyByParameterSignals` checks cooccurrence inside the loop; `detectForeignFamilyWithList` (Triage caller) defensively skips cooccurrence-required sigs since per-paramName context can't evaluate cooccurrence. Re-ingest applied to 13 affected MFRs (BACKLOG-targeted): B7 family dropped 4,317 → 1,302 (3,015 misclass products fixed). Reverse-direction bonus surfaced for Prisemi/Jingheng/YANGJIE (IGBTs misclassified as B6 in stale DB, corrected via c3 routing). Validation harness at [scripts/atlas-family-signatures-validate.mjs](scripts/atlas-family-signatures-validate.mjs) — three checks (cross-family safety, 244-IGBT non-touch, B6 dict coverage gaps); use before any re-ingest in this family-classifier space.
- **Foreign-family auto-flag system** (Decision #177): `lib/services/atlasFamilyParamSignatures.ts` declares `FAMILY_PARAM_SIGNATURES` — param names that belong unambiguously to one family (e.g. `BVCBO`/`BVCEO`/`BVEBO` → B6 BJTs, `Rds(on)` → B5 MOSFETs, `CTR`/`Viso` → E1 Optocouplers). Single source of truth feeds three consumers: (1) Triage queue route runs `detectForeignFamily(paramName, dominantFamily)` per row at render time, attaches `autoFlag: { suggestedFamily, reasoning, matchingParam }` when a foreign-family signature matches; (2) `reclassifyByParameterSignals` in `atlasMapper.ts` Phase 2 loop iterates the registry and re-routes affected products at next ingest (mirrored byte-for-byte into `scripts/atlas-ingest.mjs`); (3) future backfill script for products already in `atlas_products`. **Triage UI**: `?include=synonyms|auto_flagged|all` query param drives view-mode chip group in `TriageFilterBar`. Auto-flagged rows render with red left-border + `B1 → B6` family-transition chip + diagnosis card in place of AI translation cell + Confirm/Revert buttons (instead of Accept). Persisted status on `atlas_unmapped_param_notes` (new columns: `status`, `flagged_by`, `auto_diagnosis`) takes precedence over live registry hit — `status='wrong_family'` confirms the flag, `status='confirmed_in_family'` suppresses it (false-positive escape hatch). Sort: pending flags rise, confirmed flags sink (within auto-flagged view). Suggestion fetch is skipped for auto-flagged rows (no point asking Haiku to map a misclassified param's name). Migration in `scripts/supabase-atlas-unmapped-param-notes-schema.sql` is idempotent (DROP-then-CREATE on policies, `ADD COLUMN IF NOT EXISTS` on columns, `DO`-block constraint adds).
- **L2 category override scope** (Decision #178): `atlas_dictionary_overrides.family_id` is overloaded — accepts either an L3 familyId (`'B5'`) or an L2 category name (`'Microcontrollers'`). The inline Accept on the Triage page now uses both: `getOverrideScope(row)` returns `{ kind: 'family', key: dominantFamily }` if present, else `{ kind: 'category', key: dominantCategory }`. The ingest pipeline (`scripts/atlas-ingest.mjs`) emits a parallel `categoryCounts` alongside `familyCounts` per batch report, and the queue route aggregates per-param `dominantCategory` the same way as family. Family chip shows `L2` (info color) instead of `?` (warning) when only category is set. **Multi-category gotcha**: when the same paramName spans two L2 categories within an MFR (e.g. FMD ships MCUs + Memory with overlapping Chinese param names), the override is scoped to the dominant category only — products in the secondary category surface as still-unmapped on next regen. Workaround: clone overrides to the secondary category. Long-term fix tracked in BACKLOG (multi-category fan-out OR lift to `SHARED_PARAMS`).
- **Atlas Coverage RPC** (Decision #179): `/api/admin/atlas` GET no longer pulls full `parameters` JSONB across the wire. Instead, calls `supabase.rpc('get_atlas_coverage_aggregates', { family_attrs })` (defined in `scripts/supabase-atlas-coverage-rpc.sql`) which returns one row per `(manufacturer, family_id, category, subcategory)` tuple with `product_count`, `total_covered`, `total_rules`, `last_updated`. The route ships the family→ruleAttrs map down (~10–20 KB) and gets aggregate rows back (~5K rows, a few KB) — vs the old ~47 MB JSONB pull. Cold-cache page load drops from ~50s to ~3s. The function declares `SET statement_timeout = '300s'` to give the RPC its own budget as `atlas_products` grows. Cache layering: L1 in-memory (60s), L2 `admin_stats_cache` row keyed `'atlas-coverage'`, SWR threshold 6h. Per Decision #180, the .mjs script no longer pre-emptively deletes L2 — API route invalidate hooks own cache lifecycle, with background recompute keeping users on slightly-stale L2 instead of cold compute. Long-term BACKLOG: precomputed `coverage_attrs_count` column on `atlas_products` to eliminate per-row JSONB iteration entirely.
- **Triage Queue RPC + L1+L2+SWR cache** (Decision #180): `/api/admin/atlas/ingest/batches` GET no longer pulls `report->'unmappedParams'` JSONB across the wire. `supabase.rpc('get_triage_unmapped_aggregate')` (defined in `scripts/supabase-triage-aggregate-rpc.sql`) does the entire cross-batch dedup + MFR rollup + familyCounts/categoryCounts merge + sample-value dedup in Postgres via CTEs. Wire payload drops to ~50 KB. Cold compute: ~2-3s vs ~23s before. Cache: L1 in-memory (30 min) + L2 `admin_stats_cache` row keyed `'triage-queue'` + SWR (6h staleness threshold). Cache module: [lib/services/triageQueueCache.ts](lib/services/triageQueueCache.ts). **Two invalidate variants** (the heart of the "Sunlord doesn't appear in Triage" fix): `invalidateTriageQueueCache()` is single-flight (returns existing in-flight Promise; per-row mutations use this since optimistic UI handles freshness on the acting user); `invalidateTriageQueueCacheAndAwaitFresh()` does wait-then-restart (drains any in-flight recompute that may have read pre-commit DB state, then starts a fresh one — used by batch-state mutations: upload/proceed/revert/regenerate/batch-DELETE/proceed-all-clean). Route uses `dynamic = 'force-dynamic'` + client uses `cache: 'no-store'` to defeat any framework-level caching layered on top.

### User Context Model

Context flows at three levels, where more specific overrides more general:

- **User Profile** — Role, industry, compliance defaults, manufacturer preferences (`profiles.preferences` JSONB)
- **List Context** — Per-BOM objectives, constraints, urgency, compliance overrides (`parts_lists.context` JSONB)
- **Per-Search Context** — Family-specific application context questions (existing system, unchanged)

Global preferences produce `AttributeEffect[]` via `contextResolver.ts`, reusing the existing context modifier system. The matching engine itself doesn't change — only the effects applied to logic tables change.

The LLM adapts its conversation style and assessment focus based on user context (appended to system prompt). The UI is the same for all roles.

## FindChips Integration (Multi-Distributor Commercial Intelligence — Decision #131)

FindChips (FC) API is an aggregator covering ~80 distributors (Digikey, Mouser, Arrow, LCSC, Farnell, RS, TME, etc.) in a single ~60-200ms API call. Replaces the former per-distributor Mouser integration for pricing/stock/lifecycle. Mouser client retained solely for `SuggestedReplacement` lookups (Decision #97).

- **Client:** `lib/services/findchipsClient.ts` — API key auth (query param), 3-level cache (L1 30min, L2 Supabase 24h, L3 API), rate limiter (high safety-ceiling caps since FindChips is an internal API — defaults 1200/min, 200k/day, concurrency 20; env-tunable via `FINDCHIPS_PER_MINUTE_CAP` / `FINDCHIPS_DAILY_CAP` / `FINDCHIPS_BATCH_CONCURRENCY` — Decision #254), concurrent batch
- **Mapper:** `lib/services/findchipsMapper.ts` — Distributor name normalization, quote/lifecycle/compliance mapping, best-part selection per distributor
- **API:** `GET https://api.findchips.com/v1/fcl-search`. Env var `FINDCHIPS_API_KEY`. No distributor filter (returns all available).

**N-supplier data model:** `SupplierQuote[]`, `LifecycleInfo[]`, `ComplianceData[]` on `Part` and `EnrichedPartData`. `SupplierName` is now `string` to accommodate dynamic distributor names from FC API. `SupplierQuote` extended with `packageType`, `minimumQuantity`, `authorized`. `LifecycleInfo` extended with FC risk scores (`riskRank`, `designRisk`, `productionRisk`, `longTermRisk`).

**Pipeline integration:** Source part enriched after parts.io in `getAttributes()` via `enrichWithFindchips()`. On-demand enrichment via `/api/fc/enrich` for deferred UI enrichment.

**OEMS as conditional second source (Decision #184):** The same FC API exposes independent distributors (oemstrade.com inventory, where Chinese distributors and brokers tend to live) via `site=oems` instead of `site=fc`. We don't always hit OEMS — it'd pollute Commercial tab with broker quotes when FC has good franchised coverage. Three triggers: (1) Atlas/Chinese-MFR parts (proactive parallel fetch, keys off `mfrOrigin === 'atlas'` or `dataSource === 'atlas'` for source-side flows where origin isn't yet resolved); (2) FC empty (server-side fallback); (3) FC reports obsolete/EOL (server-side fallback). Merged with FC-wins dedup by normalized distributor name. L1 cache key is now source-keyed (`mpn::fc` / `mpn::oems`) so the two sources cache independently. Strategy enum on the client: `'fc' | 'oems' | 'parallel-both' | 'fc-with-oems-fallback'`. Default `'fc'` preserves prior behavior. No type changes, no UI tagging — quotes look identical regardless of upstream site.

**Commercial tab:** Shows one card per distributor (up to ~26). A spot-quantity control drives stock-aware ordering and a "Best @ qty N" crown (`computeBestPrice` in `bestPriceCalculator.ts`). **Ordering** (Decision #240): crowned winner first, then distributors that can fulfill the requested qty (`quantityAvailable >= qty`, cheapest unit price first), then those that can't (also cheapest first) — so out-of-stock/short-stock distributors sink. The crown goes to the cheapest distributor that can actually fulfill the qty (in-stock subset), falling back to overall-cheapest only when nobody has enough stock. **Stock highlight** (Decision #240): the In Stock cell tints light-red when `quantityAvailable < qty`; unknown stock (null) is never flagged. **Comparative crown tone** (Decision #250): the crown is colored by `comparePriceTone()` (`bestPriceCalculator.ts`) → neutral **grey** on the source part (no baseline), **green** on a replacement whose best price is lower than the source's at the same qty, light **red** when higher; tie / different-currency / missing-input → grey (no FX rate, same no-FX limitation as the crown's price sort). Wired by one prop (`CommercialContent sourcePart={sourceAttributes.part}`) on the replacement tab in `ComparisonView`; the source tab and parts-list modal pass no `sourcePart` so they default to grey. Top 5 expanded, rest behind "Show N more" toggle. Currency-aware formatting via `Intl.NumberFormat` (price sort is raw-number, no FX — same limitation as the crown). Package type and authorized badge displayed. The chat "Best Spot Price" flow (Decision #170) is separate and reports overall-cheapest, so it can diverge from the tab crown when the cheapest distributor is out of stock.

**Mouser suggestions retained (Decision #97):** `fetchMouserSuggestions()` in the recommendation pipeline still queries Mouser for `SuggestedReplacement` MPNs, injected as candidates with `certifiedBy: ['mouser']`. Env var `MOUSER_API_KEY` still required.

**Certified Cross-References (Decision #97, extended #122, #133):** External replacement suggestions from Mouser (`SuggestedReplacement`), parts.io (FFF/Functional Equivalents), and manufacturer-uploaded cross-references are unified under a `certifiedBy?: CertificationSource[]` field on `XrefRecommendation`. `CertificationSource = 'partsio_fff' | 'partsio_functional' | 'mouser' | 'manufacturer'`. All sources are resolved to full parametric candidates and scored by the matching engine. **Manufacturer cross-references** (Decision #122, bidirectional per #133): uploaded via Admin > Manufacturers > Cross-References tab (Excel/CSV with flexible column mapping), stored in `manufacturer_cross_references` Supabase table, fetched in parallel during recommendation pipeline via `fetchManufacturerCrossRefs()`. Lookup queries BOTH `original_mpn` and `xref_mpn` — reverse matches (hit on `xref_mpn`) are swapped before return so the consumer transparently treats the other side as the recommendation. MPN lookups normalize packaging suffixes via `mpnNormalizer.ts` (`-TR`, `-T/R`, `-REEL`, `-CT`, `-7INCH`, `-13INCH`, `/R7`). **Recommendation categorization** (Decision #122): Three categories — Logic Driven (blue), Manufacturer Certified (green), 3rd Party Certified (amber). `deriveRecommendationCategories()` derives from existing fields. Filter chips in RecommendationsPanel, colored category chips on each card. **Sort order** (Decision #127, extended #133, reordered #143, primary key changed #236, **reordered again #244**): the top-level key is now **Active-first** (Decision #244 — a dead part rarely belongs above a live one), then certification bucket within the lifecycle tier: Accuris Certified (parts.io FFF/functional) first, then MFR Certified, then Logic Driven (Mouser-only certified falls into Logic per bucket semantics, Decision #140). Within each group, pin-to-pin (`mfrEquivalenceType='pin_to_pin'`) sorts above functional, then by **match % / composite score** — Decision #244 replaced the Decision #236 raw-fail-count primary with match % (data-aware: missing attrs score `review` = 50% credit, so sparse crosses can't fake a 0-fail top score; raw fail-count remains a display filter only). Preferred MPN floats to top regardless of category. `sortRecommendationsForDisplay()` uses the mutually-exclusive `deriveRecommendationBucket()` so the modal ranking stays aligned with the parts-list bucket count columns. Note: the certified-cross *bypass* of quality filters (below) is unchanged — high-fail certified crosses still display, they just no longer top the list. **Certified-cross bypass** (Decision #133): `isCertifiedCross()` (true for `manufacturer` or `partsio_*`) exempts certified candidates from all 13 post-scoring blocking-rule filters (C2, C4–C10, D1–D2, E1, F1–F2) via `withCertifiedBypass()` at the call site, and from `filterRecsByMismatchCount`. An explicit human certification outranks our inferred blocking-rule rejection. **Mismatch-count post-filter** (Decision #109, extended by #159): `isCertifiedCross` / `countRealMismatches` / `filterRecsByMismatchCount` live in [lib/types.ts](lib/types.ts) as pure helpers (client-importable). A "real mismatch" is a `fail` rule where the replacement value is known and disagrees — missing-attribute fails are ignored. Certified crosses bypass; obsolete/discontinued always dropped. **Two paths:** (1) batch/list validation runs `filterRecsByMismatchCount(recs, 1)` server-side in `getRecommendations()` when `filterForBatch: true`; (2) single-part xref panel runs the filter **client-side** in `RecommendationsPanel` at threshold `≤2`, with a "Show all" toggle (`hideHighFails` state, default true) under the filter popover's Quality section + dismissible "Showing high-fail" chip in the filter row. Server-side single-part path returns the full candidate set so the toggle is instant and doesn't require a refetch.

**Columns:** `fc:lifecycle`, `fc:riskRank`, `fc:designRisk`, `fc:productionRisk`, `fc:longTermRisk` (Risk & Lifecycle group). `commercial:bestPrice`, `commercial:totalStock` (Commercial summary, auto-aggregate across all N supplier quotes). Old `mouser:*` columns removed; saved views auto-strip via `sanitizeTemplateColumns()`.
