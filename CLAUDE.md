# XRefs App

An AI-powered component intelligence platform for the electronics industry. Users enter a part number (or upload a BOM), and the system finds equivalent replacements, scores them with a deterministic rule engine, and adapts its recommendations based on the user's role, objectives, and business constraints.

**Current MVP:** Cross-reference recommendation engine covering 43 passive, discrete semiconductor, IC, frequency control, optoelectronic, and relay families, with Digikey as the primary data source.

**Vision:** A platform that helps engineers, buyers, and supply chain professionals make better component decisions by combining deep technical matching with commercial intelligence, compliance and lifecycle awareness, and supply chain insights вҖ” personalized to each user's context. See `docs/PRODUCT_ROADMAP.md` for the full vision and phased implementation plan.

## Tech Stack

- **Next.js 16** (App Router, Turbopack) + React 19 + TypeScript
- **MUI v7** with Material Design 3 dark mode (`colorSchemes` API, `enableCssLayer: false`)
- **Anthropic Claude API** (Sonnet 4.6 default, overridable via `ANTHROPIC_MODEL`) for LLM orchestrator with tool calling
- **Supabase** for auth, user profiles, conversation persistence
- **Digikey Product Information API v4** for live part data (OAuth2 client credentials)
- **Emotion** for CSS-in-JS

## How the Matching Engine Works

The core pipeline is in `lib/services/partDataService.ts вҶ’ getRecommendations()`:

1. **Get source attributes** вҖ” Fetch the original part's parametric data (Digikey API, mock fallback)
2. **Merge overrides** вҖ” Apply any user-supplied attribute corrections
3. **Classify family** вҖ” Map subcategory string вҶ’ family ID, detect variant families (e.g., current sense within chip resistors) via `lib/logicTables/familyClassifier.ts`
4. **Get logic table** вҖ” Load the family's matching rules from `lib/logicTables/`
5. **Apply admin overrides** вҖ” `overrideMerger.ts` fetches active DB overrides and merges onto the TS base (removeвҶ’overrideвҶ’add pattern). Patchable fields include `valueAliases` (Decision #160) for per-rule categorical synonym groups.
6. **Apply context** вҖ” If user answered application context questions, `contextModifier.ts` adjusts rule weights/types. Context questions are also subject to admin overrides.
7. **Fetch candidates** вҖ” Search Digikey for replacement candidates using critical parameters as keywords
8. **Score candidates** вҖ” `matchingEngine.ts` evaluates each candidate against every rule
9. **Apply qualification-domain filter** (Decisions #155, #196, broadened by #211) вҖ” After scoring, classify source + every candidate into a `QualificationDomain` (automotive_q200 / automotive_q101 / automotive_q100 / medical_implant / mil_spec / space / ...) via `qualificationDomain.ts`. Hard-exclude cross-domain candidates (medical_*/mil_spec/space under automotive) **before** the certified-cross bypass so certified crosses can't route around the gate. Phase 1.5 (#196): also hard-excludes `commercial` and `industrial_harsh` candidates under automotive intent. `unknown` candidates stay visible вҖ” most Atlas products and non-classified MFRs lack the AEC attribute even when AEC-qualified, so excluding `unknown` would wipe out legitimately-qualified parts. Classifier registry still Phase 1 (Murata MLCCs only) вҖ” expanding to TDK / Samsung / Yageo / KEMET / AVX / Kyocera / Taiyo Yuden tracked in BACKLOG. **Attribute-based upgrade** (Decision #211): `upgradeFromAttributes` now checks `aec_q200`/`aec_q101`/`aec_q100` parametric attributes AND the `part.qualifications[]` string array (populated uniformly by `extractQualifications` across all Digikey categories вҖ” load-bearing for C1 LDOs / C6 Voltage Refs and other categories where the per-category param map doesn't carry AEC fields).
10. **Apply post-scoring family filters** вҖ” `withCertifiedBypass` applies per-family blocking-rule filters (C2/C4вҖ“C10/D1вҖ“D2/E1/F1вҖ“F2); bypassed for MFR-certified and Accuris-certified crosses (Decision #133). **Automotive AEC enforcement is now table-driven** (Decision #221, full rollout June 3, 2026): `AUTOMOTIVE_AEC_ENFORCEMENT` table in `partDataService.ts` enrolls all 11 AEC-aware families (B5/B6/B7/B8/B9 AEC-Q101, C9/C10 AEC-Q100, D1/D2/F1 AEC-Q200, E1 AEC-Q101). Each row carries `{ familyId, questionId, answerValue, attributeId }` because family files don't agree on a single context questionId вҖ” B-block + C-block use `automotive`, D1 uses `extended_temp_automotive`, D2/F1 use `automotive_aec_q200`, E1 uses `automotive_aec_q101`. One generic `filterAutomotiveAecMismatches(recs, src, ctx, familyId)` consumes the table; one generic `applyContextSourceOverrides` iterates the table and synthesizes `aec_q*: 'Yes'` onto the scoring-local source so the identity_flag rule fires. E1 and F1 chain вҖ” the new AEC filter runs alongside the existing `filterOptocouplerMismatches` / `filterRelayMismatches` (independent concerns, order-independent). Adding a new family is a one-row addition; no new code paths.
11. **Sort** вҖ” `sortRecommendationsForDisplay` sorts by **Active-first (top-level key, Decision #244)**, then certification bucket (Accuris вҶ’ MFR вҶ’ Logic) within the lifecycle tier, then mfr-equivalence rank, qualification domain (`context-matched > unknown > deviation`), then **match % / composite score**. Decision #244 made Active the top key (a dead part rarely belongs above a live one вҖ” fixes obsolete crosses topping the panel) and replaced the Decision #236 raw-fail-count primary with match %, which is data-aware: a missing attribute scores `review` = 50% credit (vs `fail` = 0), so a sparse candidate is dragged toward 50% and can't masquerade as a flawless 0-fail match the way `countRealMismatches` let it (it ignored `review` entirely). Raw fail-count survives only as a display filter (`filterRecsByMismatchCount`), never as a sort key.

### Scoring

Each rule has a **weight** (0-10). The matching engine evaluates each rule and produces:
- `matchPercentage = (earnedWeight / totalWeight) * 100`
- A part **fails** if any rule result is `'fail'` (actual value mismatch only вҖ” missing data never causes fail)
- Missing candidate data вҶ’ `'review'` (flagged for human verification, never rejected). `blockOnMissing` flag controls note severity only, not the result (Decision #109).
- `application_review` rules get 50% credit (can't be automated)
- `operational` mismatches get 80% credit (non-electrical)

## QC & Feedback System

The QC page (`/qc`) is a top-level admin-only route (sidebar icon: `RateReviewOutlinedIcon`). It is **feedback-first**: the primary section shows user-submitted feedback items for triage, not raw recommendation logs.

### Architecture

- **Feedback Tab** (primary): Lists all `qc_feedback` rows with status filters (Open/Reviewed/Resolved/Dismissed), search, and sort. Clicking a row opens a detail view that reconstructs the comparison table from the JSONB snapshot stored in `recommendation_log`, highlighting the specific rule the user flagged.
- **Logs Tab** (secondary): Lists raw `recommendation_log` entries with feedback indicators. Has **Export** (CSV/JSON) and **AI Analysis** capabilities.
- **Recommendation Logger** (`lib/services/recommendationLogger.ts`): Writes snapshots (source attributes, recommendations, context Q&A, overrides) to `recommendation_log` as JSONB. Must always be `await`ed вҖ” fire-and-forget silently drops entries.
- **Feedback types**: `FeedbackStage` = `rule_logic` | `qualifying_questions`. `FeedbackStatus` = `open` | `reviewed` | `resolved` | `dismissed`.
- **Comparison reconstruction**: The feedback detail view reuses the row-building algorithm from `ComparisonView.tsx` вҖ” builds rows from `sourceAttributes.parameters` + `matchDetails`, highlights the flagged rule with a yellow left border.

### Export & AI Analysis (Logs Tab)

- **Export** (`/api/admin/qc/export`): CSV (flat columns + top 3 recs) or JSON (full snapshots) download. Respects current filters, caps at 10K rows. Client uses `window.open(url)` for direct download.
- **AI Analysis** (`/api/admin/qc/analyze`): Server-side aggregation (`lib/services/qcAnalyzer.ts`) computes per-family/per-rule stats from JSONB snapshots, sends compact summary to Claude Sonnet 4.5 via streaming API. SSE events forward text deltas to `QcAnalysisDrawer` (right-side MUI Drawer with `ReactMarkdown` rendering).
- **Aggregation approach**: Raw snapshots are too large for Claude (~5-10K tokens each). `aggregateQcStats()` computes rule failure rates, missing attribute frequency, score distributions, and feedback correlations, then includes 3-5 representative examples. Total input ~3-5K tokens regardless of log count.

### QC Component Structure

`QcShell.tsx` (`components/qc/`) is the top-level orchestrator with settings toggle in the header and `QcSectionNav` for Feedback/Logs sections. Content components live in `components/admin/`:
- `QcFeedbackTab.tsx` вҶ’ `QcFeedbackDetailView.tsx` (feedback flow)
- `QcLogsTab.tsx` вҶ’ inline DetailView + Export/Analyze buttons (logs flow)
- `QcAnalysisDrawer.tsx` вҖ” AI analysis right-side drawer with streaming markdown
- Shared: `QcFeedbackCard.tsx`, `QcRecommendationSummary.tsx`, `qcConstants.ts`

## App Feedback (Collaborative)

Separate from QC feedback. App feedback is the general user-facing channel вҖ” bugs, ideas, anything not tied to a specific recommendation rule. Two-way thread between the submitting user and the platform admin (Rob). Decisions #162 (initial submission flow + attachments), #220 (two-way thread + comments + read state), #222 (Trello-style overlay UX + activity-first sort + delete + RLS fix).

**Data model:**
- `app_feedback` вҖ” original submission + status (`open`/`reviewed`/`wip`/`resolved`/`dismissed`; `wip` = Work In Progress, added June 9 2026, ordered between reviewed and resolved вҖ” migration `scripts/supabase-app-feedback-add-wip-status.sql` updates the CHECK constraint on existing tables) + JSONB `attachments` + `user_last_read_at` + `admin_last_read_at`. The only way a non-admin can stamp `user_last_read_at` вҖ” RLS blocks direct UPDATE for users (intentional, so they can't tamper with status/notes). Both the GET-thread route and the comment-POST route call this RPC. See [[silent-update-under-rls]] memory.
- Storage bucket `app-feedback-attachments`: per-user folder prefix, signed-URL reads (1h TTL), user can delete own, admin can delete any.

**Surfaces (user side at `/feedback`):**
- `FeedbackShell` вҖ” full-width card list + "+ New Feedback" header button + 30s background poll (paused when tab hidden, refreshes on `visibilitychange`).

Mounts вҶ’ fires GET-thread вҶ’ server stamps the appropriate `*_last_read_at` вҶ’ modal echoes `onUpdated({})` so the parent list can flip `hasUnread` locally and clear the visual signals immediately.

- `FeedbackRowMenu` (shared) вҖ” kebab вҶ’ "Delete thread" вҶ’ confirm dialog вҶ’ `DELETE /api/app-feedback/[id]` (cleans storage paths first, then DB row; comments cascade). Confirm copy explicitly spells out that delete affects both sides.

**Surfaces (admin side, `/monitoring вҶ’ AppFeedbackTab`):**
- Table-based list (paginated). Default `sort_by=activity` вҖ” server fetches the full filtered set, sorts `(pinned desc, hasUnread desc, created_at desc)`, slices to the page. Column-header sort overrides activity until refresh.

Created under the admin's own `user_id`; on submit, the list resets to page 0 and reloads so it surfaces at top.

## Replacement Preferences (Decisions #145, #146)

Per-list configuration stored on `parts_lists.replacement_priorities` JSONB column, shape `ReplacementPriorities` in `lib/types.ts`. Editable via List Settings вҶ’ Replacement Preferences tab (`ReplacementPrioritiesField.tsx`). Two kinds of inputs:

- **Scoring inputs** (affect composite score + cache variant): `order: ReplacementAxis[]` (priority ranking of Lifecycle / Compliance / Cost / Stock), `enabled: Record<ReplacementAxis, boolean>`. Changes trigger `handleRefreshRows`. Composite score computed in `getRecommendations()` via `computeCompositeScore()` (lib/services/compositeScore.ts), stored on `XrefRecommendation.compositeScore`. Used as within-bucket tiebreak: match % floor for Logic within Вұ2%, composite primary for certified buckets. Stock axis is gated вҖ” only active when source `totalStock < 100`.
- **Display filters** (client-side, not cache-varied): `hideZeroStock?: boolean`, `maxReplacements?: 1-5`, `buckets?: RecommendationBucket[]`, plus **`preferBuyable?: boolean` + `buyableRequires?: 'price' | 'price_and_stock'` (Decision #260)**. Applied in `PartsListTable.tsx` via `pickEffectiveTopRec()` + `recPassesFilters()` + `getAlternateReplacements()`. No refresh triggered on toggle вҖ” `handleUpdateListDetails` separates `rankingChanged` (only `order`/`enabled`) from filter-only changes. Legacy `suggestionBuckets` / `maxSuggestions` keys are still accepted on read (pre-Decision-#147) and progressively migrate on next save.
- **Prefer-buyable #1 pick** (Decision #260, default ON): when the best-matching #1 replacement has no live distributor availability, `pickEffectiveTopRec()` promotes the closest *buyable* candidate (the demoted certified cross stays visible as an alternate). "Buyable" = has a distributor price (`'price'`) or price **and** stock (`'price_and_stock'`, default), per `isBuyable()` in `lib/services/buyableSelection.ts` (pure, unit-tested). Reach is the row's persisted candidate pool вҖ” `replacement` + alternates + `cheapestViableRecs` (the last is FC-enriched in `runFCEnrichment` when the pref is on, near-free since MPNs overlap). Display-only, instantly reversible, no cache bump; never hides (a row with no buyable cross keeps its pick) and never overrides an explicit `preferredMpn`.
- **рҹҮЁрҹҮі flag on Chinese (Atlas) MFRs in the list table** (Decision #261): `MfrWithFlag` helper in `PartsListTable.tsx` appends the same flag+tooltip as the recommendation cards. Repl. MFR keys off `topRec.part.mfrOrigin === 'atlas'` (covers alternates); source MFR (`mapped:manufacturer`/`dk:manufacturer`) keys off `row.resolvedPart?.dataSource === 'atlas'` (mfrOrigin isn't resolved for source rows in the list flow).

**Three-layer defense for `hideZeroStock`:** (1) render-time swap picks first stocked candidate from persisted `[replacement, ...replacementAlternates]`; (2) post-FC-enrichment promotion effect persists the swap; (3) deep-fetch effect (concurrency 2) handles rows where all persisted top-3 are zero-stock вҖ” calls `getRecommendations` + `enrichWithFCBatch(top-30)` because `getRecommendations` returns candidates with empty `supplierQuotes` by design (FC enrichment deferred). `deepFetchAttemptedRef` prevents infinite retry on cohorts with no stocked alternative.

**Persisted alternates cap:** bumped 2 вҶ’ 4 in `toStoredRows` (both Supabase + localStorage) to support `maxReplacements: 5`. Existing lists progressively migrate on next write.

**Repl. Distributor column (`sys:top_suggestion_supplier`):** reads `supplierQuotes[0].supplier` (pre-sorted by best unit price). Uses `SUPPLIER_DISPLAY` from `AttributesTabContent.tsx`. Column ID preserved pre-rename; header label is "Repl. Distributor".

**Vocabulary (Decision #147):** Parts-list code uses "replacement" throughout. `PartsListRow.replacement` = the top proposed swap (singular); `PartsListRow.replacementAlternates` = positions #2вҖ“#5. Storage layer reads legacy `suggestedReplacement` / `topNonFailingRecs` keys for back-compat. Do NOT confuse with `LifecycleInfo.suggestedReplacement` вҖ” that's Mouser's manufacturer-published successor MPN on EOL parts, a genuinely different concept.

**Column picker MPN vs SKU:** `dk:mpn` (Product Identity, label "MPN (DK)") surfaces `EnrichedPartData.mpn` вҖ” canonical MPN as resolved by source, distinct from user's raw `mapped:mpn`. `dk:digikeyPartNumber` renamed to "DigiKey SKU" and moved to Commercial group вҖ” it's a distributor SKU, not part identity.

**Testing routes (Decision #277).** Until July 2026 this repo had 1,631 tests and **not one executed an API route** вҖ” while every write to a parameter mapping happens in a route. There is now a harness in [`__tests__/helpers/`](__tests__/helpers/) (zero new dependencies, `testEnvironment: 'node'`, no jsdom): `supabaseMock.ts` (stateful, supports **writes**, records filters and op order, injects per-table/per-op errors), `authGuard.ts`, `routeHarness.ts`. Two traps: `jest.mock()` paths must be **RELATIVE** (`../../lib/...`; the `@/` alias resolves for `import` but not inside `jest.mock`), and `lib/supabase/server` + `lib/supabase/service` must be pointed at **one shared instance via `globalThis`** or a cross-table assertion is two unrelated hopes. A write returns `data: null` unless `.select()` is chained вҖ” the mock models that deliberately. The mock does NOT model unique indexes, RLS, FK/CHECK, Postgres collation, the 1000-row cap, or concurrency; those belong to [docs/QA_PARAM_MAPPING.md](docs/QA_PARAM_MAPPING.md).

**The bar for these suites is mutation, not green.** Break the source one line at a time and the suite must catch it. Every mapping-write route, the override read layer and the real ingest mapper were built this way; it repeatedly found gaps review had not (documented contracts nothing enforced, a filter the stub silently ignored, thin fixtures). A suite that only passes proves nothing вҖ” see the `green-test-must-fail-on-broken-code` memory.

**`scripts/atlas-ingest.mjs` is importable but still a standalone CLI.** An entrypoint guard (`IS_CLI`) means importing it does not parse argv, build a Supabase client, or run the dispatcher; it exports `mapModel` / `classifyAtlasCategory` / `cleanManufacturerName` **for tests only**. Any change here must keep `--report --dry-run` and the no-argument usage output byte-identical (stdout, stderr, exit code). The golden file pins the mapper's real output over real fixture models: regenerate with `npm run atlas:golden` **and read the diff** вҖ” it names, in attribute names, exactly what your change did to product data.

**No parameter value is ever discarded** (Decision #278). When two source spellings resolve to one `(family, attributeId)` slot the first wins вҖ” but the loser is **kept under its raw parameter name** via `storeRawValue()`, never dropped. This is not a nicety: accepting a mapping in Triage used to *delete* the value it displaced (a weight-9 `rds_on` rule was left holding a *typical* number where the *maximum* had been available), and 534,819 values across 437,093 products were being binned this way. Three details are load-bearing and each is pinned by a mutation-tested case: the raw key **keeps Unicode letters/digits** (an ASCII-only slug reduces a pure-Chinese name to `''` and drops the value вҖ” that alone was 251,643 values, and it also gave Triage an empty `attributeId`); **identical** values are still dropped (nothing to preserve, and the comparison is deliberately conservative вҖ” a false "differs" costs a redundant copy, a false "same" deletes data); and colliding keys are **suffixed, never overwritten** (`MAX_LOSER_SUFFIX = 200` is a runaway guard, *not* a data cap вҖ” a bound of 20 dropped 1,530 real values). The first occurrence keeps its historical key, so no ingested row re-keys **unless it collides with a scoring slot** (see below). вҡ пёҸ The rule exists in **both** copies and they are pinned against each other by the parity block in [__tests__/services/atlasLayer1KeepLoser.test.ts](__tests__/services/atlasLayer1KeepLoser.test.ts) вҖ” a "keep in lockstep" comment is not a mechanism, and note the parity block must compare **KEYS as well as values**: a normalized winner (`"<30 V"` вҶ’ `"30"`) and a rescued loser (`"30"`) produce an identical value set whether the loser was preserved or deleted. Layer 1 stops a wrong mapping *destroying* data; it does not make the mapping *right* вҖ” several semantically wrong mappings it exposed (power вҶ’ `color`, current вҶ’ `supply_voltage`) are in BACKLOG.

**An unrecognised parameter's number must still be scaled вҖ” but ONLY when its unit is certain** (Decision #280, supersedes the storage half of #279). `matchingEngine.ts:23` returns a stored `numericValue` **verbatim, with no re-parse and no unit check**, and the rescued path (`storeRawValue`, for a column no dictionary recognises) used bare `extractNumeric` вҖ” it read the number and ignored the unit. Siliup/SP40N25TQ ships `еҜјйҖҡз”өйҳ»(RDS(on)) = "80mО©@10V"`, which slugs onto `rds_on` (weight 9, `lte`) and was stored as **80, not 0.08**. Demonstrated end-to-end: a genuinely better 80 mО© candidate hard-*failed* against a 100 mО© source; it now passes rated better. Measured over all 429 files / 437,387 products: **153,993 values corrected, 50,521 of them in scoring slots, across 69,639 products.** The fix is ONE line in `storeRawValue` in both mapper copies вҖ” `rescueNumericValue`, which scales **only** when the unit is on a conservative allowlist shared verbatim via [lib/services/atlas-rescue-units.json](lib/services/atlas-rescue-units.json) (`{p n u Вө Ој m k K M G} Г— `{F A V W H Hz s О© J C S}`, minus the 8 excluded ambiguous tokens incl. `Gs`=gauss). вҡ пёҸ **The allowlist IS the safety property.** Both "proper" fixes were measured and rejected: a first-letter rule reads `ppm` as pico (11,122 values) and `pcs` as pico (11,254); a strict whole-token rule instead BREAKS `PF` (1,989), `nV/вҲҡHz` (474), `mW/sr` (391), `Mbit` (431). There are **367 distinct unit spellings** here вҖ” recognising them all is its own project. An unrecognised token keeps today's value EXACTLY, so this can only make a number more correct or leave it alone. вҡ пёҸ But the TABLE is not self-validating вҖ” the cross generated `Gs` (gauss, not giga-second) and mis-scaled 229 values, so ANY change to `prefixes`/`atoms` must re-run the corpus-occurrence audit (`atlasRescueUnits.test.ts`) before it can be trusted. Uppercase `P`/`U`/`N` are absent for the same reason (enabling `PF` generatively also enables `UV`/`PW`). **Scope: the rescued path only** вҖ” the dictionary-mapped paths already normalise and are untouched. Four gates ran over the full corpus and must stay at zero for any change here: **keys added / keys removed / display strings changed / numbers lost.**

**#279's reserved-id RENAME was withdrawn вҖ” do not reintroduce it** (Decision #280). #279 moved an unvetted value to a new key (`rds_on` вҶ’ `raw_rds_on`) so it could not be scored. But **the key is a storage contract**: four consumers find values by it вҖ” the Specs panel (`fromParametersJsonb`), the coverage RPC, the widening RPC (in SQL, where `raw_` has no meaning) and the reclassify RPC вҖ” and only two were updated. Measured: the rename changed **14,110 key names across 6,559 products (16%)**, dropped coverage corpus-wide, leaked `raw_` labels to users, and freed the old slot so a *different* measurement could occupy it (verified live: YANGJIE/FRD60A600AS-290A, vendor `VRRM=600` displaced by dict-mapped `VR (V)=630`). It also demoted CATEGORICALS (`channel_type = "N"`), which makes matching **more permissive, not safer** вҖ” a missing value scores `review` = 50% credit that never fails, so an N-channel source could surface P-channel parts. вҡ пёҸ An `unvetted: true` annotation was designed as the replacement and **also rejected**, on two verified grounds: `fromParametersJsonb` builds its output as an explicit field-by-field literal, so an added field is silently dropped and never reaches the engine; and the mapped write is an *unconditional* overwrite, so a naive annotation would have **destroyed** the value the rename at least preserved. The real lesson: **the wrong-scale number was the whole bug** вҖ” an unlabelled column named `RDS(on)` landing in `rds_on` is usually correct, and once its scale is right it scores correctly.

## External API Access (Decision #80)

The matching engine is accessible to sister products and external systems via two paths:

**REST API** вҖ” Existing endpoints (`/api/search`, `/api/attributes/{mpn}`, `/api/xref/{mpn}`) accept `Authorization: Bearer <key>` for machine-to-machine auth. Keys configured in `XREFS_API_KEYS` env var. See `docs/API_INTEGRATION_GUIDE.md` for full documentation.

**MCP Server** вҖ” Standalone stdio-transport server for AI agent integration. 5 tools: `search_parts`, `get_part_attributes`, `find_replacements`, `list_supported_families`, `get_context_questions`. Run with `npm run mcp:dev`. Test with `npm run mcp:inspect`. See `mcp-server/` directory.

**Auth architecture:** `requireAuth()` in `lib/supabase/auth-guard.ts` checks API key first, then falls back to Supabase cookie auth. API key users get a fixed service user ID; admin routes remain blocked. `lib/supabase/server.ts` has a try/catch fallback that returns a direct Supabase client when `cookies()` from `next/headers` is unavailable (MCP server, standalone scripts).
